# pi-extension-template

Template repository for [pi](https://github.com/anthropics/pi-coding-agent) extensions — pick a layout and start building.

## Directory Layouts

Choose one of three structures based on your project's complexity:

### Layout 1: Bare file (NOT recommended)

```
my-extension.ts
```

A single `.ts` file at the repo root. Simple but has significant drawbacks:

- ❌ No `package.json` — no version tracking, no metadata
- ❌ No README — no documentation
- ❌ Cannot be independently installed via `pi install`
- ❌ Hard to maintain as the extension grows

**Verdict:** Only suitable for quick experiments. Do not use for published extensions.

### Layout 2: Simple plugin (recommended for single-plugin repos)

```
my-plugin/
├── index.ts          # Extension entry point
├── package.json      # Plugin metadata + version
└── README.md         # Plugin documentation
```

Each plugin is a self-contained directory at the repo root. Clean and straightforward.

**Best for:** A repo containing one or a few independent plugins (e.g., provider collections).

**Example in this template:** [`example-provider/`](./example-provider)

### Layout 3: Grouped plugins (recommended for complex repos)

```
category-name/
└── my-plugin/
    ├── index.ts
    ├── package.json
    └── README.md
```

Plugins are organized under category directories. Useful when plugins are related or designed to work together.

**Best for:** A suite of related tools/skills, or a large collection that benefits from categorization.

**Example in this template:** [`tools/example-plugin/`](./tools/example-plugin)

### Comparison

| | Layout 1 (bare) | Layout 2 (simple) | Layout 3 (grouped) |
|---|---|---|---|
| Version tracking | ✗ | ✓ | ✓ |
| Documentation | ✗ | ✓ | ✓ |
| `pi install` support | ✗ | ✓ | ✓ |
| Complexity | Minimal | Low | Medium |
| Recommended | ✗ | ✓ | ✓ |

## Quick Start

1. **Fork / clone this template**

   ```bash
   git clone git@github.com:YOUR_USERNAME/pi-extension-template.git my-extensions
   cd my-extensions
   ```

2. **Choose your layout** (Layout 2 or 3 recommended)

3. **Copy an example** and rename it

   ```bash
   # Layout 2
   cp -r example-provider my-new-plugin
   # Layout 3
   cp -r tools/example-plugin tools/my-new-plugin
   ```

4. **Edit the files** — replace placeholder values with your actual implementation

5. **Update root `package.json`** — add your plugin directory to `pi.extensions`

6. **Test locally**

   ```bash
   pi -e ./my-new-plugin/index.ts
   ```

7. **Generate docs & release**

   ```bash
   npm install
   npm run update-docs
   bash scripts/release.sh my-new-plugin patch
   ```

See [AGENTS.md](./AGENTS.md) for the complete publishing guide.
